一个总结

遇见途虎

今天是2019年8月4日，在途虎的第345天，印象之初，途虎是一个正处于迅猛发展中的互联网信息化平台，很荣幸见证了它的发展壮大，并成为其中的一员。

四段经历

业务综合-仓库组

第一个阶段，做的业务是门店货权转移，解决商品交易过程中，门店和途虎的利润分成，商品货主，库存变更问题。

这项工作的难点是和我交接的只有一份文档。要完成这份工作，首先要梳理代码逻辑，完善现有的文档，画出系统流程调用链路，其中每一个不确定的业务点和产品确认，每一行代码逻辑和对接方反复推敲，在对业务和代码逻辑有一定理解后完善代码，测试过程中及时和测试人员配合，最终达成交付。(:ps 有老板盯着，大家处理速度配合效率好高哦

物流组

货权这份工作顺利上线后，时间线刚好是我的试用期过了，由于物流组比较缺人，就被调到物流组。

期间优化了倒包裹接口，通过线程分组，mq拆接口的方式性能提升百倍。

升级老的windows服务拉单系统到 .Net core。

  • 业务上画出流程交互图，整理和快递公司的对接文档，涉及到顺丰，圆通，中通，韵达，邮政EMS等知名物流公司的接口对接。
  • 技术上将单号库从数据库sqlserver迁移到mysql，以及解决服务化系统后带来的分布式并发问题。

订单组

服务化快递拉单系统后，被调到了业务订单组，主要做了两件事情：

  • 地图，用于电销人员下单时选择门店
    由于老式的地图下单页面非常卡顿，技术架构陈旧，已经远远不能满足目前途虎电销人员的需要，为了提高生产力，我们决定重构订单地图系统。

    • 前端：LayUI + Bootstrap + Jquery + Html + Ajax + Css
    • 后端：.Net
    • DB层：Mysql + Redis + ES

  • 监控，用于实时监控途虎电话系统的进线信息

    从前端到后台独立重构了途虎呼叫中心监控系统，搭建了一套后台配置管理体系，实现监控页面的所有数据都可以配置化管理。

交易中台

当业务体量达到一定程度后，系统间交互，服务治理变成一件很头痛的问题，由此诞生了交易中台。

19年7月1日，被调到了交易中台项目组，技术栈也从此转向了Java，我知道，新的旅途要开始了。

一点感想

我非常感谢这份经历，因为经历是最宝贵的财富。

希望接下来可以超出预期的做些事情，感觉还没有达到自己想要的样子，可以再挖掘挖掘。

愿景,计划,准备

愿景
  • 明年这个时候薪水翻倍
  • 参加开源社区，提供更好的数字化服务给这个世界
计划

  • Spring实战，Thinking in Java，Spring boot编程思想

  • Spring boot源码解析

  • 参与开源社区项目，保持Watch，如Spring Cloud Alibaba

  • IO,集合，多线程

  • 分布式，redis，消息，ES

  • leetcode

    • 注意事项：保持专注，切勿今天研究分布式，明天研究ES，什么都想学往往最后都不深入
准备

  • 读书

    • 选择读自己比较感兴趣的章节
    • 根据目录，简读一遍，知道每一章在讲什么，解决什么问题
    • 再读一遍，这一遍还是通读，但是重点部分要深读（：当天整理博客，思维导图的时候会用到

  • 参与开源社区

    • 先看别人的源码解读，然后自己看+debug，写出源码解析
    • 可以不看别人的源码解读，写出源码解析文章，画出架构图
    • 持续watch代码，看下别人怎么解决issue，commit的
    • 提issue，参与讨论，改bug
    • 贡献文档，代码
    • 帮助别人解决问题

以终为始，这是我的故事，希望能对有缘看到这篇文章的人有所启发。

最后用八个字作结束：坚持，学习，总结，坚持。
